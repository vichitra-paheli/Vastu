## Overview

`PageRegistry` is a singleton that holds all statically-defined pages for your application. You register pages in `apps/<your-app>/src/pages.ts`, which the framework loads at app startup.

Static pages (registered in code) and dynamic pages (created via builder mode and stored in the database) are merged. Static pages always appear first in the sidebar, followed by dynamic pages separated by a subtle divider. In builder mode, static pages show a "System" badge — they cannot be deleted at runtime, but their configuration can be overridden.

## Registering a page

```typescript
// apps/my-app/src/pages.ts
import { PageRegistry } from '@vastu/workspace';

PageRegistry.register({
  id: 'products',
  name: 'Products',
  icon: 'ShoppingBag',
  template: 'table-listing',
  section: 'PAGES',
  order: 1,
  config: {
    table: 'Product',
    columns: [
      { field: 'name', header: 'Name' },
      { field: 'sku', header: 'SKU' },
      { field: 'price', header: 'Price', displayType: 'currency' },
    ],
  },
});
```

## `PageRegistration` type

| Property | Type | Required | Default | Description |
|----------|------|----------|---------|-------------|
| `id` | `string` | Yes | — | Stable unique identifier. Used as the Dockview panel type ID when the page opens. Must be unique across all registered pages. |
| `name` | `string` | Yes | — | Display name shown in the sidebar, command palette, and page picker. |
| `template` | `PageTemplate` | Yes | — | Template type that renders the page. See [Template values](#template-values). |
| `icon` | `string` | No | `'LayoutDashboard'` | Tabler icon name without the `Icon` prefix. Browse at [tabler.io/icons](https://tabler.io/icons). |
| `section` | `PageSection` | No | `'PAGES'` | Sidebar section. See [Sidebar sections](#sidebar-sections). |
| `order` | `number` | No | `100` | Sort order within the section. Lower numbers appear first. Pages with the same order are sorted alphabetically. |
| `config` | `Record<string, unknown>` | No | `{}` | Initial page configuration passed to the template. Shape depends on the template. |

## Template values

| Value | Template | Use for |
|-------|----------|---------|
| `'table-listing'` | TableListingTemplate | Tables with filters, sort, pagination, export |
| `'summary-dashboard'` | SummaryDashboard | KPI cards + charts, a single entity overview |
| `'multi-tab-detail'` | MultiTabDetail | Tabbed detail view for a single record |
| `'data-explorer'` | DataExplorer | Interactive chart explorer with dimension/metric controls |
| `'form-page'` | FormPage | Multi-step form / wizard |
| `'timeline-activity'` | TimelineActivity | Chronological event stream |
| `'dashboard'` | Dashboard | Free-form dashboard with pinnable cards |
| `string` | custom | Any custom template string your app registers |

## Sidebar sections

Pages are grouped into sidebar sections and rendered in section order: PAGES → SYSTEM → ADMIN.

| Value | Intended use |
|-------|-------------|
| `'PAGES'` | Primary domain pages (most pages go here) |
| `'SYSTEM'` | Framework or admin utilities |
| `'ADMIN'` | Admin-only pages |

## Registry methods

The `PageRegistry` singleton is exported from `@vastu/workspace`. These methods are available:

```typescript
import { PageRegistry } from '@vastu/workspace';

// Register a page (throws if ID already exists)
PageRegistry.register(registration: PageRegistration): void

// Get all pages sorted by section → order → name
PageRegistry.getAll(): PageRegistration[]

// Get pages for one sidebar section
PageRegistry.getBySection(section: PageSection): PageRegistration[]

// Look up a single page by ID
PageRegistry.getById(id: string): PageRegistration | undefined

// Check if an ID is registered
PageRegistry.has(id: string): boolean

// Count of registered pages
PageRegistry.size: number
```

`unregister(id)` and `clear()` are exported for test isolation. Do not call them in production code.

## Registering multiple pages

```typescript
import { PageRegistry } from '@vastu/workspace';

const pages = [
  {
    id: 'drivers',
    name: 'Drivers',
    icon: 'User',
    template: 'table-listing' as const,
    section: 'PAGES' as const,
    order: 1,
    config: { table: 'Driver' },
  },
  {
    id: 'constructors',
    name: 'Constructors',
    icon: 'Car',
    template: 'table-listing' as const,
    section: 'PAGES' as const,
    order: 2,
    config: { table: 'Constructor' },
  },
  {
    id: 'driver-profile',
    name: 'Driver Profile',
    icon: 'UserCircle',
    template: 'multi-tab-detail' as const,
    section: 'PAGES' as const,
    order: 10,
  },
];

for (const page of pages) {
  PageRegistry.register(page);
}
```

## Error behavior

Registering a page with a duplicate ID throws synchronously:

```
Error: [PageRegistry] Page "products" is already registered.
Each page ID must be unique. If you intend to override, call unregister() first.
```

This is intentional — duplicate registrations in `src/pages.ts` are a programming error. The error surfaces at app startup, before any user interaction.

## How pages load at startup

When the workspace initializes, it imports `src/pages.ts`. That import runs the `PageRegistry.register()` calls as side effects. The sidebar and command palette then read from `PageRegistry.getAll()` to build their lists.

You do not need to configure anything else. The framework wires up the import automatically via `apps/<app-name>/src/app/workspace/page.tsx`.
