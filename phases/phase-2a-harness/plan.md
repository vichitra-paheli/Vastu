## US-202: Template Data Source Integration

### Subtask 202-1: FilterNode to Prisma Where Translator

**Components to create or modify:**
- NEW `packages/shared/src/utils/prisma-filter.ts` — pure function: `filterNodeToPrismaWhere(node: FilterNode, allowedColumns: string[]): Prisma.JsonObject`
  - Include → `{ column: { in: values } }`
  - Exclude → `{ NOT: { column: { in: values } } }`
  - Regex → `{ column: { contains: pattern, mode: 'insensitive' } }`
  - Composite AND/OR → nested `{ AND: [...] }` / `{ OR: [...] }`
  - Column allowlist validation (prevent injection via column names)
- MODIFY `packages/shared/src/index.ts` — re-export

**State management:** None (pure function in shared package)

**Testing strategy:**
- Unit tests: every FilterNode type × every filter mode → expected Prisma where clause
- Unit tests: nested AND/OR groups
- Unit tests: invalid column names rejected
- No DB needed — pure function tests

**Design system mapping:** Implements Patterns Library §2.4 (FilterNode spec)

**Estimated complexity:** M (150 lines + 200 lines tests)

### Subtask 202-2: Data Query API Route

**Components to create or modify:**
- NEW `packages/shell/src/app/api/workspace/data/query/route.ts`
  - GET: accepts params: table, columns (JSON array), filters (FilterNode JSON), sort (JSON array), page, pageSize, search
  - Translates to Prisma: `findMany` with `where`, `orderBy`, `skip`, `take`
  - Returns `{ rows, total, columns: ColumnMetadata[] }`
  - Global search: OR across string columns with `contains` (AC-5)
  - Related data joins: include driver/constructor names on race_results (AC-7)
  - Query timeout: 10 seconds (AC-8)
  - Tenant scoping via session.user.organizationId (AC-8)
- NEW `packages/shell/src/app/api/workspace/data/query/__tests__/route.test.ts`

⚠️ HUMAN DECISION NEEDED: The `table` parameter maps to a Prisma model name. We need a whitelist of allowed table names to prevent unauthorized model access. Proposed whitelist: `['season', 'circuit', 'constructor', 'driver', 'race', 'raceResult', 'qualifyingResult', 'sprintResult', 'pitStop', 'lapTime', 'driverStanding', 'constructorStanding', 'raceEvent', 'penalty']`.

**API surface:** `GET /api/workspace/data/query?table=...&filters=...&sort=...&page=...&pageSize=...&search=...`

**Estimated complexity:** L (350 lines)

## US-208: Server-Sent Events for Workspace

### Subtask 208-1: SSE Endpoint

**Components to create or modify:**
- NEW `packages/shell/src/app/api/workspace/events/route.ts` — SSE endpoint using Next.js `ReadableStream`
  - Tenant-scoped: only sends events for the user's organization (AC-3)
  - Event types: `record.created`, `record.updated`, `record.deleted`, `view.saved`, `config.changed` (AC-2)
  - Heartbeat every 30 seconds to keep connection alive
  - Auto-reconnect support via `retry:` field in SSE stream

No new npm dependencies — uses native ReadableStream per technical constraints.

**Estimated complexity:** M (200 lines)

### Subtask 208-2: SSE Event Bus (Server-Side)

**Components to create or modify:**
- NEW `packages/shell/src/lib/event-bus.ts` — in-process event emitter for workspace events
  - `emitEvent(orgId, event)` — publishes to all connected SSE streams for that org
  - `subscribe(orgId, callback)` / `unsubscribe` — used by SSE route handler
  - Uses Node.js EventEmitter pattern (no Redis pub/sub yet — single-process for Phase 2A)

⚠️ HUMAN DECISION NEEDED: For production multi-process deployment, this would need Redis pub/sub. For Phase 2A (single-process demo), in-process EventEmitter is sufficient. Document this limitation in ADR.

**Estimated complexity:** S (80 lines)

### Subtask 208-3: SSE Client Hook and UI Integration

**Components to create or modify:**
- NEW `packages/workspace/src/hooks/useSSE.ts` — client-side SSE connection hook

## Phase-Level Concerns

### Cross-Cutting Issues

1. **PageConfiguration migration to Prisma**: Currently stored in-memory. Must be migrated to a Prisma table (`page_configurations`) so that seeded demo page configs persist across restarts. This affects US-202 (data source config), US-203 (seeded page configs), and US-205 (chart data sources).

2. **FilterNode shared between packages**: The `FilterNode` type exists in both `packages/shared/src/types/view.ts` and `packages/workspace/src/components/FilterSystem/types.ts`. The `filterNodeToPrismaWhere` utility must use the shared package type. Ensure these types stay aligned.

3. **Table name to Prisma model mapping**: A whitelist mapping table parameter names to Prisma delegate names is needed across US-202, US-206, and US-207. Centralize this in `packages/shared/src/utils/prisma-tables.ts`.

4. **Seed determinism**: All generators in US-201 must use the same seeded PRNG instance (passed through, not re-initialized per generator) to ensure globally deterministic output.

5. **Breaking change: SidebarNav data source**: US-203 changes SidebarNav from mock data to database-backed pages. This affects the sidebar across the entire workspace.

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Lap time seed performance (65K rows) | Medium | Medium | Use `createMany` with batch inserts; set 45s timeout in test |
| CASL conditions to Prisma translation edge cases | Medium | High | Extensive unit tests on filterNodeToPrismaWhere; start with simple cases |
| SSE connection stability in development | Low | Low | Exponential backoff already planned; add heartbeat |
| PageConfiguration migration conflicts | Medium | Medium | Create migration early (US-200); coordinate with existing in-memory store |
| Chart keyboard nav complexity across 5 chart types | Medium | Medium | Start with line/bar; extend to donut/scatter as separate PR |

### HITL Summary

1. **New database table**: `page_configurations` — needed to persist demo page configs seeded in US-203. Not explicitly in requirements but implied by existing PageConfiguration type.
2. **New CASL resources**: Adding F1-specific resources (Race, Driver, etc.) to the permissions model (US-207).

4. **SSE architecture**: In-process EventEmitter sufficient for Phase 2A single-process demo, but will need Redis pub/sub for production multi-process deployment (US-208).

## Architecture Decision Records

### ADR-001: Data Pipeline Architecture

See `/docs/decisions/ADR-001-data-pipeline.md` (to be created in US-212).

**Decision**: Use Prisma as the sole query engine for the data pipeline. FilterNode trees are translated to Prisma `where` clauses via a pure function in `packages/shared`. No raw SQL.

**Rationale**: Prisma provides type safety, migration management, and relation handling. The FilterNode-to-Prisma translator is a pure function that can be unit-tested without a database. This aligns with the codebase convention "All database access via Prisma. No raw SQL."

**Alternatives considered**:
- Raw SQL for complex aggregations (rejected: violates codebase conventions)
- GraphQL layer (rejected: premature abstraction, adds complexity without matching the FilterNode model)

### ADR-002: SSE over WebSocket

See `/docs/decisions/ADR-002-sse-over-websocket.md` (to be created in US-212).

**Decision**: Use Server-Sent Events (SSE) via Next.js ReadableStream for real-time updates. No WebSocket.

**Rationale**: SSE is simpler (HTTP/1.1 compatible, auto-reconnect built into EventSource API, no additional libraries needed). The data flow is unidirectional (server → client) which is the SSE sweet spot. Client-to-server communication is handled by standard API routes. Next.js App Router supports SSE natively via ReadableStream. WebSocket would require a separate server process and additional infrastructure.

**Trade-offs**: SSE has a 6-connection limit per domain in HTTP/1.1 (not an issue with HTTP/2). SSE does not support binary data (not needed). For Phase 3+ agent streaming, SSE may need to be augmented.

### ADR-003: PageConfiguration Persistence

⚠️ HUMAN DECISION NEEDED

**Context**: PageConfiguration is currently stored in an in-memory Map (`_stores.ts`). For Phase 2A, seeded demo page configurations must persist across server restarts.

**Options**:
- **Option A** (recommended): Add `page_configurations` table to Prisma schema. Migrate existing in-memory store to Prisma. Seed configs alongside other F1 data.
- **Option B**: Seed configs via API calls in a startup script. Keep in-memory store. Configs are re-seeded on every server start.
- **Option C**: Use a JSON file on disk as a config cache. Seed writes to the file; the API reads from it.

**Recommendation**: Option A. Aligns with the codebase convention of using Prisma for all data persistence and makes configs queryable, auditable, and properly backed up.
